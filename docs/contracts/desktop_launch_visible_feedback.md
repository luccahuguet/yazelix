# Desktop Launch Visible Feedback

## Summary

Desktop entry launches must invoke `yzx desktop launch` directly from the desktop shell. Yazelix records launch evidence under the terminal launch log directory so a disappearing desktop activation can be diagnosed after the fact.

## Why

GNOME/Wayland desktop activation can route `Terminal=true` entries through host starter terminals such as `xdg-terminal-exec`, Ptyxis, or Ghostty. Those starter terminals can exit and clean up their process scope before Yazelix writes launch evidence or before the managed Kitty/Zellij window survives.

The supported contract is that Linux desktop entries avoid the starter terminal and run `yzx desktop launch` with `Terminal=false`. `yzx desktop launch` owns runtime resolution, config parsing, generated terminal config repair, Ghostty cursor rerolls, and the deferred handoff to the packaged Yazelix terminal. Launch diagnostics are durable logs plus `yzx doctor`, not a host starter terminal window.

## Scope

- User-local desktop entries created by `yzx desktop install`
- Linux Home Manager desktop entries generated by the Yazelix module
- `yzx desktop launch` pre-terminal progress and failure behavior
- Doctor diagnostics for stale desktop entries that still route through host starter terminals

## Behavior

- Managed Yazelix desktop entries must set `Terminal=false` so host starter terminals cannot kill the Yazelix handoff.
- User-local and Linux Home Manager desktop entries must share the same desktop-entry contract.
- `yzx desktop launch` must print concise progress before invoking the fast path.
- If the fast path succeeds, `yzx desktop launch` must schedule the managed Yazelix terminal through the deferred desktop launcher.
- Desktop fast-path terminal children must skip the interactive welcome keypress gate and proceed to Zellij directly. GUI desktop activation is already the user's launch confirmation; stopping before Zellij makes the workspace keymap unavailable and turns terminal focus/key-event quirks into a startup blocker.
- The deferred desktop launcher must write non-empty schedule evidence before detaching so failed starter-terminal handoffs are diagnosable instead of leaving zero-byte logs.
- On Linux hosts with a user systemd manager, the deferred desktop launcher should detach through a transient user scope before falling back to `nohup`/`setsid`, because some starter terminals clean up their whole descendant scope when the starter window exits.
- If launch fails before terminal handoff, `yzx desktop launch` must print the actionable error to stdout/stderr and leave durable evidence for `yzx doctor`.
- The desktop path must preserve runtime re-anchoring and inherited-environment cleanup. It must not trust stale ambient Yazelix, Zellij, Yazi, or shell activation state.
- Doctor must report desktop entries that still point at the expected launcher but set `Terminal=true`, because that routes through the unsupported starter-terminal path.

## Non-goals

- Building a custom graphical prelaunch dialog
- Changing terminal package selection
- Requiring host starter terminals for visibility
- Adding a second fallback launch path after the desktop fast path fails
- Moving desktop-specific UX into config parsing or terminal config generation
- Solving graphical failure-dialog UX

## Acceptance Cases

1. When `yzx desktop install` writes the user-local desktop entry, the entry sets `Terminal=false` and still points at the expected launcher.
2. When Home Manager generates Yazelix desktop entries on Linux, they also set `Terminal=false`.
3. When `yzx desktop launch` fails before terminal handoff, the error is printed and the launch path leaves durable evidence for doctor.
4. When `yzx desktop launch` succeeds, the managed terminal command is spawned by Yazelix from a detached helper.
5. When the managed terminal child starts from `yzx desktop launch`, it receives the one-shot welcome skip flag so Zellij starts without a pre-Zellij keypress prompt.
6. When desktop handoff cannot complete, the launch log is non-empty and names the attempted detachment method.
7. When an installed desktop entry points at the expected launcher but sets `Terminal=true`, `yzx doctor` reports it as stale with repair guidance.
8. Desktop launch still clears inherited Yazelix, Zellij, and Yazi session variables before invoking the runtime fast path.

## Open Questions

- Should a future release add a purpose-built graphical failure dialog for desktop-launch failures that occur before the packaged terminal can be spawned, replacing the starter terminal window?
- Should doctor eventually validate the exact rendered Home Manager desktop entry from a full Home Manager evaluation rather than only the installed profile entry contract?
